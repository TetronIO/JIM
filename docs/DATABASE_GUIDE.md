# JIM PostgreSQL Database Guide

> Configuration, connection pooling, backup/restore, and environment-specific notes for JIM's PostgreSQL database.

---

## Configuration Overview

JIM uses PostgreSQL 18 with environment-specific tuning. Compose files are layered — later files override earlier ones:

| Layer | File | Purpose |
|-------|------|---------|
| Base | `docker-compose.yml` | Production/deployment defaults (64GB/16c target) |
| Dev overrides | `docker-compose.override.yml` | Tracked dev settings (ports, env, LANG, conservative DB) |
| Local tuning | `docker-compose.local.yml` | **Gitignored** — auto-generated machine-specific DB tuning |
| Standalone DB | `db.yml` | Simple DB for local development (`jim-db`) |
| Standalone tuning | `db.local.yml` | **Gitignored** — auto-generated tuning for standalone DB |

The `*.local.yml` files are generated by `.devcontainer/postgres-tune.sh` during initial setup and can be regenerated with `jim-postgres-tune`.

> **Note:** The auto-tuning reserves a portion of system RAM for JIM services, VS Code, and build tools before allocating the rest to PostgreSQL. This balance may need adjusting as JIM's memory profile evolves — particularly if JIM.Worker processes very large datasets or if application-level caching is introduced. See `.devcontainer/POSTGRES_TUNING.md` for monitoring guidance and how to adjust the split.

## Full Stack Settings

The `docker-compose.yml` configuration is optimised for **64GB Windows / 32GB WSL / 16 cores / SSD** using [PGTune](https://pgtune.leopard.in.ua/):

```

shared_buffers=8GB
effective_cache_size=24GB
maintenance_work_mem=2GB
checkpoint_completion_target=0.9

```

## Connection Pooling

JIM uses Npgsql connection pooling with the following default settings:

| Parameter | Value | Description |
|-----------|-------|-------------|
| Minimum Pool Size | 5 | Keep connections warm for common operations |
| Maximum Pool Size | 30 | Per-service limit (3 services × 30 = 90 max, leaving headroom) |
| Connection Idle Lifetime | 300s | Recycle idle connections after 5 minutes |
| Connection Pruning Interval | 30s | Check for idle connections every 30 seconds |

> **Note:** JIM has 3 services (Web, Worker, Scheduler). The Maximum Pool Size must leave headroom within PostgreSQL's `max_connections` for superuser connections, monitoring tools, and ad-hoc sessions. The default 30 per service uses 90 of the 200 available connections, leaving 110 for headroom.

### Recommended Pool Sizes by Deployment Size

| Environment | Services | Max Pool/Service | Total Max | PostgreSQL max_connections |
|-------------|----------|------------------|-----------|----------------------------|
| Development | 3 | 20 | 60 | 100-200 |
| Small (< 10k objects) | 3 | 30 | 90 | 200 |
| Medium (10k-100k objects) | 3 | 50 | 150 | 200-300 |
| Large (100k+ objects) | 3-6 | 50 | 150-300 | 300-500 |

### Monitoring Connection Pool

Enable Npgsql logging by setting the logging level to `Debug` for the `Npgsql` logger. Connection pool statistics will appear in logs during high activity.

## Backup and Restore

### Creating a Backup

Using Docker:
```bash
# Backup to a timestamped file
docker exec jim.database pg_dump -U ${DB_USERNAME} -Fc ${DB_NAME} > jim_backup_$(date +%Y%m%d_%H%M%S).dump

# Or use plain SQL format (larger but human-readable)
docker exec jim.database pg_dump -U ${DB_USERNAME} ${DB_NAME} > jim_backup_$(date +%Y%m%d_%H%M%S).sql
```

### Restoring from Backup

```bash
# Restore from custom format (.dump)
docker exec -i jim.database pg_restore -U ${DB_USERNAME} -d ${DB_NAME} --clean < jim_backup.dump

# Restore from SQL format
docker exec -i jim.database psql -U ${DB_USERNAME} -d ${DB_NAME} < jim_backup.sql
```

### Scheduled Backups (Production)

For production deployments, consider:
1. Using a volume-mounted backup directory
2. Setting up cron jobs for automated backups
3. Implementing backup rotation (keep last N backups)
4. Testing restore procedures regularly

## Environment-Specific Notes

### Auto-Tuning (All Environments)

PostgreSQL is automatically tuned during devcontainer setup. The script (`.devcontainer/postgres-tune.sh`) detects available CPU cores and RAM, then generates `docker-compose.local.yml` and `db.local.yml` with optimal [PGTune](https://pgtune.leopard.in.ua/) settings for an OLTP workload.

To re-tune after changing devcontainer resources:
```bash
jim-postgres-tune
jim-db-stop && jim-db        # local dev
jim-restart                  # Docker stack
```

### Tracked Defaults (Conservative)

The tracked files (`docker-compose.override.yml`, `db.yml`) contain conservative defaults suitable for the smallest supported dev environment (~8GB). The auto-tuned local overlays replace these with machine-optimal values.

If the local overlay files don't exist (e.g., first clone without running setup), the conservative defaults are used — PostgreSQL will work, just not optimally.
