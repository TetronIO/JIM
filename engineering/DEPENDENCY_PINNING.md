# Dependency Pinning Policy

This document is the canonical, holistic policy for how JIM pins, verifies, and updates every third-party dependency it consumes, from NuGet packages through to base container images. It consolidates policy that was previously scattered across `DEVELOPER_GUIDE.md`, `COMPLIANCE_MAPPING.md`, `.github/dependabot.yml`, and the individual pin-check workflows; those documents now point here for the overall picture and keep only their layer-specific detail.

## Purpose and threat model

JIM is deployed in high-trust environments (healthcare, finance, government) and is designed to be **self-contained and air-gap deployable**. Two consequences follow directly:

- **Supply chain integrity.** A dependency (direct or transitive) that is silently substituted, whether by a compromised registry, a typosquat, or an unpinned version resolving differently between two builds, is a code-execution path into every JIM deployment. This maps to OWASP Top 10:2025 [A03 (Software Supply Chain Failures) and A08 (Software or Data Integrity Failures)](plans/doing/OWASP_TOP_10_ASSESSMENT.md), assessed in full in the OWASP Top 10:2025 assessment.
- **Reproducible, auditable builds.** Given the same commit, a build today and a build in a year must resolve to the exact same set of dependency versions, so a customer (or JIM's own release pipeline) can prove what shipped. Floating version ranges and unpinned transitive graphs defeat this.

The working principle: **every dependency JIM's production artefacts depend on is pinned to an exact, known version, with CI mechanically enforcing the pin rather than relying on developer discipline.** Where a layer is deliberately pinned more loosely (docs tooling, integration test images), that is a documented, reviewed exception, not an oversight.

## Inventory

| # | Layer | Mechanism | Enforcement | Update path |
|---|-------|-----------|--------------|--------------|
| 1 | NuGet direct packages | Exact versions in `.csproj` (no floating ranges) | Human review of every Dependabot PR before merge | Dependabot, weekly, grouped by ecosystem area (`.github/dependabot.yml`); major versions blocked, reviewed manually |
| 2 | NuGet transitive packages | `packages.lock.json` per project + `RestoreLockedMode` | `RestoreLockedMode` forced whenever `CI=true` (`Directory.Build.props`); explicit `--locked-mode` / `/p:RestoreLockedMode=true` in CI, release, and container image builds **(new; this change)** | Regenerated by `dotnet restore --force-evaluate` when a direct version changes; automated for Dependabot PRs by `regenerate-nuget-lock-files.yml` |
| 3 | Production container base images | `FROM image:tag@sha256:<digest>` in every production Dockerfile | CI `discover-base-images` job fails any production Dockerfile missing a digest; `# jim-compliance: production-image` directive marks scope | Dependabot `docker` ecosystem, one grouped PR per Dockerfile |
| 4 | OS (apt) packages inside images | Exact `pkg=version` pins in Dockerfile `RUN apt-get install` | Not visible to Dependabot or to the digest-scoped Trivy scan; `apt-pin-check` workflow raises a bump PR when a newer, installable version exists in the base image's archive | `apt-pin-check` (daily), validated installable before proposing |
| 5 | GitHub Actions | 40-character commit SHA pins with a `# vX.Y.Z` version comment | Convention, checked at PR review; `tooling-pin-check` does **not** cover Actions (it covers `@playwright/mcp` and `dotnet-ef`, pinned outside any Dependabot-readable manifest; see that workflow's header) | Dependabot `github-actions` ecosystem, weekly; `codeql-action` sub-actions grouped so init/autobuild/analyze move together |
| 6 | Devcontainer features/images | Exact feature versions in `devcontainer.json`, resolved digests captured in `devcontainer-lock.json` | Convention; a Feature bump needs a full devcontainer rebuild to validate | Dependabot `devcontainers` ecosystem, weekly |
| 7 | Compose infrastructure images | PostgreSQL digest-pinned in `docker-compose.yml`; Keycloak pinned to a minor version tag in `docker-compose.override.yml` | Convention. Keycloak's tag pin is a **deliberate, softer pin**: it is dev-adjacent SSO infrastructure (not shipped in the product image), and a minor-tag pin still gets security patches within that minor line without a PR per patch release | Dependabot `docker-compose` ecosystem, daily, scoped via `exclude-paths` |
| 8 | Docs tooling (mkdocs, via pip) | Bounded version ranges (e.g. `mkdocs>=1.6,<2`) in `.devcontainer/setup.sh` and the docs publish workflows (`docs.yml`, `publish-api-reference.yml`, `release.yml`) | Accepted: docs tooling runs only at authoring/publish time, is never part of the shipped product, and a range already absorbs patch/minor releases | Reviewed periodically; upper bound is a manual judgement call, out of scope for `tooling-pin-check` |
| 9 | Integration test images | Mutable tags in `test/integration/docker/` compose files | Accepted: test-only, never shipped, and images are chosen for connector compatibility testing rather than supply chain assurance | Updated manually as needed |

Rows 3-6 pin by digest, SHA, or exact version and are all Dependabot-covered; rows 4-5 additionally have a bespoke bot (`apt-pin-check`, `tooling-pin-check`) closing a gap Dependabot's ecosystems cannot see, as documented in each workflow's own header comment. Row 2 (NuGet transitive pinning) is the gap this change closes.

## Verification and provenance

Pinning stops substitution at build time; the following controls verify what is already pinned and provide tamper evidence downstream:

- **Trivy base-image scanning** (`scan-base-images` CI job) scans every production base image on every push and PR, failing on fixable HIGH/CRITICAL CVEs. Suppressions are governed by `.trivyignore`: every entry requires a CVE ID, affected component, justification, and a ~3-month review date (see root `CLAUDE.md` > Security).
- **SBOM generation, cosign signing, and SLSA provenance attestation** run at release time (`.github/workflows/release.yml`), so a shipped image's exact dependency graph and build provenance are independently verifiable after the fact.

## Governance

Adding a **new** NuGet package or other third-party dependency is a separate process from updating a pinned version: it requires explicit user approval before it is added. See root `CLAUDE.md` > "Third-Party Dependency Governance" for the full notify-research-present-approve flow and the maintainer preference order (Microsoft-maintained > established corporate-backed > .NET Foundation > well-maintained OSS with identifiable maintainers).

## Operations

**When a locked-mode restore fails locally** (a direct package version changed and the lock file has not caught up):

1. Run `dotnet restore JIM.sln --force-evaluate` to regenerate the affected `packages.lock.json` file(s) against the current `.csproj` graph.
2. Review the diff: confirm only the expected package(s) moved, and check the new transitive versions for anything unexpected.
3. Commit the regenerated lock file(s) alongside the version bump.

**When a Dependabot NuGet PR fails locked-mode restore in CI:** this is expected on the first CI run and self-heals. Dependabot does not reliably regenerate `packages.lock.json` across project references when it bumps a `.csproj` version (dependabot-core#12318, dependabot-core#10863), so its PR branch initially has a stale lock file. The `regenerate-nuget-lock-files.yml` workflow triggers on the push to the `dependabot/nuget/**` branch, runs `dotnet restore --force-evaluate`, and pushes a signed `chore: regenerate NuGet lock files for Dependabot update` commit back to the same branch via the GitHub GraphQL `createCommitOnBranch` mutation (signed because `main`-bound branches require signed commits; pushed with a GitHub App installation token, not `GITHUB_TOKEN`, because only App-authored pushes re-trigger the required CI checks). That second CI run is green. If it is not, either the workflow failed (check its run log) or the drift is a real, unexpected transitive resolution change that needs investigating rather than papering over.

If the regeneration workflow itself fails (for example, a required secret is missing or NuGet resolution genuinely cannot satisfy the graph), regenerate manually with the same command used above and push a normal commit; the automation is a convenience, not the only path.

See `.claude/skills/review-dependabot/SKILL.md` for what a reviewer checks on a NuGet Dependabot PR, including the lock-file regeneration commit.

**JIM.Web pins `RuntimeFrameworkVersion` to match the production base image.** `Microsoft.NET.Sdk.Web` projects (only `src/JIM.Web/JIM.Web.csproj` in this solution) add an implicit `Microsoft.AspNetCore.App.Internal.Assets` entry to their lock file, whose version otherwise floats to whichever ASP.NET Core shared runtime patch is installed on the machine performing the restore. `src/JIM.Web/JIM.Web.csproj` pins `<RuntimeFrameworkVersion>` to the exact ASP.NET Core version bundled by the pinned `mcr.microsoft.com/dotnet/aspnet`/`dotnet/sdk` base image digests, so the lock file entry is deterministic everywhere. **Update this pin whenever those base image digests are bumped** to a new patch version, or the JIM.Web lock file will drift from what the production image actually ships (`dotnet restore JIM.sln --force-evaluate` after updating the pin regenerates it; verify with `dotnet build`/`dotnet test` plus a JIM.Web Docker build). Separately, `src/JIM.Web/Dockerfile`'s standalone `dotnet restore` step (a layer-caching optimisation only) deliberately omits `RestoreLockedMode`: a known upstream bug (dotnet/aspnetcore#64897) causes an isolated single-project `dotnet restore` of a `Sdk.Web` project to drop the implicit framework-reference package from its locked-mode comparison and fail regardless of what the lock file contains. The subsequent `dotnet publish` step (which performs its own implicit restore) is unaffected and is what actually enforces the lock file for JIM.Web.
