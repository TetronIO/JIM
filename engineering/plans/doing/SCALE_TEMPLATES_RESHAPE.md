# Scale Templates Reshape

- **Status:** Doing (Phases 1-4 and 6 complete; Phase 5 population validation done, deeper performance characterisation pending)
- **Issue:** [#741](https://github.com/TetronIO/JIM/issues/741)

## Overview

Rename the five `Scale*` integration test templates to encode their group count in the template name (e.g. `Scale100K` becomes `Scale100k50Groups`), and add a new `Scale100k5kGroups` template that provides a realistic long-tail group distribution representative of a 100,000-person enterprise. The new template is OpenLDAP-only and Scenario 8-only initially.

## Business Value

### Problem

The current `Scale100K` template has **50 groups** for **100,000 users**. This was reduced from an originally planned 2,040 groups because `samba-tool group addmembers` holds an LDB write lock per call, making millions of membership writes impractical against Samba AD. The reduction was a workaround, not a deliberate design choice (documented in [`Test-Helpers.ps1:219-222`](../../../test/integration/utils/Test-Helpers.ps1#L219-L222) and [`Test-GroupHelpers.ps1:118-121`](../../../test/integration/utils/Test-GroupHelpers.ps1#L118-L121)).

Now that OpenLDAP support is implemented and OpenLDAP does not share Samba's per-call locking bottleneck, we can offer a more representative dataset. A real 100,000-person organisation typically has 5,000-15,000 groups distributed as a long tail, not 50 large groups.

Two further problems compound this:

1. The template name `Scale100K` does not convey its group count. With multiple group-count variants becoming possible, contributors need to read the populator scripts to know which shape they are about to provision.
2. The current per-scenario tests against Scale100K are not exercising realistic group-membership topology (many small groups, few large groups, varied types and persona categories), which is where production sync churn lives.

### Goals

- Make the scale ladder self-describing through naming
- Provide a representative group-shape test for Scenario 8 (Cross-Domain Entitlement Sync), the scenario most sensitive to realistic entitlement topology
- Preserve all existing Samba and OpenLDAP test coverage at every scale tier
- Avoid silent template behaviour divergence between directory types

### Non-Goals

- Reshaping the 200K-1M tier group models (deferred to a follow-up review; the renames in this plan only acknowledge their current group counts)
- Extending the new long-tail model to Scenarios 1, 2, 3, 7, or 9 (re-evaluated later based on Scenario 8 validation results)
- Changes to user-side data shape (user counts, attributes, departments) at any tier

## Technical Architecture

### Current State

Scale templates are defined in two PowerShell helper functions, both supporting the same set of names:

- `Get-TemplateScale` in [`test/integration/utils/Test-Helpers.ps1`](../../../test/integration/utils/Test-Helpers.ps1) returns `Users`, `Groups`, `AvgMemberships` per template. Consumed by [`Populate-SambaAD.ps1`](../../../test/integration/Populate-SambaAD.ps1) and [`Populate-OpenLDAP.ps1`](../../../test/integration/Populate-OpenLDAP.ps1) (used by Scenarios 1, 7, 9).
- `Get-Scenario8GroupScale` in [`test/integration/utils/Test-GroupHelpers.ps1`](../../../test/integration/utils/Test-GroupHelpers.ps1) returns category-aware counts (`Companies`, `Departments`, `Locations`, `Projects`) per template. Consumed by Scenario 8 populators.

Current Scale100K shape via `Get-Scenario8GroupScale`:

| Category | Count | Member rule | Size at 100k users |
|---|---|---|---|
| Company | 5 | All users for that company | ~20k each |
| Department | 10 | All users for that department | ~10k each |
| Location | 5 | 30-50% of users, capped at 50k | ~30k-50k each |
| Project | 30 | Tier-cycled 50-5,000 | Long tail |
| **Total** | **50** | | **~450k memberships** |

Group types cover Security/Distribution × Universal/Global/DomainLocal × mail-enabled/not (six combinations), generated by `Get-GroupTypeDistribution` with a percentage-based bucket scheme.

Template names appear in `ValidateSet` attributes across many scripts, in two interactive menu functions in `Run-IntegrationTests.ps1`, in snapshot image tags, in cached CSV filenames, and in documentation.

### Proposed Solution

**Template name changes** (mechanical sweep):

| Current | Renamed |
|---|---|
| `Scale100K` | `Scale100k50Groups` |
| `Scale200K` | `Scale200k55Groups` |
| `Scale500K` | `Scale500k65Groups` |
| `Scale750K` | `Scale750k70Groups` |
| `Scale1M` | `Scale1m80Groups` |

Casing convention: lowercase scale suffix (`k`, `m`) after the numeral, mirrored on group count (`Groups`).

**New template** (`Scale100k5kGroups`):

Roughly 5,000 groups distributed as a long tail. Indicative shape:

| Category | Count | Typical size | Total membership contribution |
|---|---|---|---|
| All-staff | 2 | 80,000-100,000 | ~180,000 |
| Division | 10 | 5,000-25,000 | ~150,000 |
| Location | 15 | 1,000-10,000 | ~75,000 |
| Department | 100 | 200-3,000 | ~150,000 |
| Application access | 500 | 10-300 | ~100,000 |
| Project/team | 3,000 | 5-30 | ~60,000 |
| Distribution list | 1,000 | 10-150 | ~80,000 |
| Role | 400 | 20-500 | ~120,000 |
| **Total** | **~5,027** | | **~900k memberships (~9 per user)** |

Exact numbers will be tuned during implementation. The shape goal is: a handful of very large groups (retaining the existing top-end stress test), a moderate count of medium-sized groups, and a long tail of small groups (5-50 members each), which is where realistic enterprise sync churn lives.

Three new categories beyond the current four (`Company`, `Department`, `Location`, `Project`): `Application`, `DistributionList`, `Role`. `Company` is retired as a distinct concept; the new `Division` slot fills its role for very-large groups, and `AllStaff` covers truly company-wide groups.

**Directory support for the new template:**

- OpenLDAP only
- Samba AD selection raises a hard error at parameter validation time, with a clear message pointing users at `Scale100k50Groups` or smaller tiers
- The error fires in the interactive menu (before scenario selection), the `-Template` parameter validation (before population starts), and the Scenario 8 populator entry points (defence in depth)

**Scenario support for the new template:**

- Scenario 8 only
- Other scenarios that currently accept `Scale100K` (1, 7, 9) will accept the renamed `Scale100k50Groups` but not `Scale100k5kGroups`
- Mis-use is caught at scenario entry with a clear error: "Scale100k5kGroups is only valid for Scenario 8"

### Data Flow

No data flow changes. The new template is a different parameter value that selects a different group shape in `Get-Scenario8GroupScale` and (potentially) a new long-tail-aware helper function. Population still flows: scenario script -> populator script -> `samba-tool`/`ldapadd`/`ldapmodify` -> directory.

### Phase 2: Long-Tail Group Model ✅

**Goal:** Implement the new group shape in helper functions, ready for use by Scenario 8 populators.

**Changes:**

1. Extend `Get-Scenario8GroupScale` with a `Scale100k5kGroups` entry, with the category counts shown in the table above.
2. Extend `New-TestGroup` to accept three new categories: `Application`, `DistributionList`, `Role`. Each gets its own naming convention (e.g. `App-{Name}`, `DL-{Name}`, `Role-{Name}`), description text, and managedBy distribution.
3. Add a `Get-LongTailGroupSize` helper that returns a target member count following a power-law distribution, parameterised by category. Replaces the fixed `sizeTiers` array currently used for Projects.
4. Extend `New-Scenario8GroupSet` to produce groups for the new categories when the template defines them.
5. Add reference name pools: division names (curated, ~15 entries), application names (mix of common SaaS-style names), role names (RBAC-style), distribution list names (department/interest patterns).
6. Adjust `Get-GroupTypeDistribution` if the new categories warrant different default type/scope/mail-enabled choices (e.g. role groups likely Security/Universal/no-mail; distribution lists Distribution/Universal/mail-enabled).

**Validation:** Unit-test the helpers by invoking `New-Scenario8GroupSet -Template Scale100k5kGroups` and asserting:
- Total group count is between 4,500 and 5,500
- At least one group per category
- Membership size distribution follows expected ranges per category
- All groups have unique names

### Phase 3: Populator Extension ✅

**Goal:** Make the OpenLDAP Scenario 8 populator handle ~5,000 groups and ~1M memberships within the existing time budget.

**Changes in `Populate-OpenLDAP-Scenario8.ps1`:**

1. Membership assignment loop must handle long-tail categories (`Application`, `DistributionList`, `Role`) in addition to existing four. Each category gets its membership rule:
   - `Application`: random subset of users (independent of department/company)
   - `DistributionList`: random subset weighted toward department/location overlap
   - `Role`: random subset of users matching title-pattern filters where possible (e.g. all Managers)
2. Batch `ldapmodify` across multiple groups per `docker exec` call rather than one per group. Target batch size ~5,000 `member:` lines per call; expected ~150-200 calls total for ~1M memberships.
3. Consider embedding initial members directly in the initial `ldapadd` LDIF for small groups (< 100 members) rather than separate `add: member` operations.

**Changes in `Populate-SambaAD-Scenario8.ps1`:**

Add a guard at script entry:

```powershell
if ($Template -eq "Scale100k5kGroups") {
    throw "Template Scale100k5kGroups is not supported on Samba AD. Use Scale100k50Groups for Samba scale testing, or switch to -DirectoryType OpenLDAP."
}
```

Same guard mirrored in `Run-IntegrationTests.ps1` parameter validation and the interactive menus so users discover the constraint at selection time, not 30 minutes into population.

**Validation:** Full population run against OpenLDAP Scale100k5kGroups; measure wall-clock time and confirm under 15 minutes for population alone. Compare end-state directory contents (group count, total memberships, category distribution) against expected values.

### Phase 4: Scenario Restriction ✅

**Goal:** Restrict `Scale100k5kGroups` to Scenario 8.

**Changes:**

Add scenario-level validation at the entry point of each non-Scenario-8 scenario script:

```powershell
if ($Template -eq "Scale100k5kGroups") {
    throw "Template Scale100k5kGroups is only valid for Scenario 8 (Cross-Domain Entitlement Sync). Use Scale100k50Groups or smaller for this scenario."
}
```

Apply to: `Invoke-Scenario1-HRToIdentityDirectory.ps1`, `Invoke-Scenario7-ClearConnectedSystemObjects.ps1`, `Invoke-Scenario9-PartitionScopedImports.ps1`. Scenarios 2 and 3 are Samba-only so the Samba guard already covers them. Scenarios 4, 5, 6 hard-code their templates so are not affected.

`Run-IntegrationTests.ps1` interactive menu adjusts to grey out or hide `Scale100k5kGroups` when a non-Scenario-8 scenario is already selected.

**Validation:** Try running each non-Scenario-8 scenario with `Scale100k5kGroups`; confirm clear error before any work starts.

### Phase 5: JIM-Side Validation Run (Partial)

**Goal:** Characterise the performance and correctness profile of the new template against JIM.

**Status:**

- Population side fully validated end-to-end. First run produced 4,932 groups instead of 5,027 and revealed four bugs in the OpenLDAP populator (Department cap at `$sortedDepartmentKeys.Length`, Location cap at the hardcoded inline list, plus uniform-size Division and Location memberships from modulo bucketing rather than the tier table). Bugs fixed in a follow-up commit; second run produced the expected 5,027 groups with the full long-tail bucket distribution.
- Deeper performance characterisation (per-phase wall-clock, worker memory peak, DB row counts at each stage, comparison against `Scale100k50Groups`) **not yet captured** — to be tracked separately.

**Activities completed:**

- Full population run against OpenLDAP. Source and Target both reached 5,027 groups and 751,768 memberships per CS; sync replication produced identical state on both directories.
- Long-tail buckets verified: groups <50 members = 3,698; 50-199 = 1,022; 200-999 = 232; 1,000-4,999 = 49; 5,000-19,999 = 22; 20,000-49,999 = 2; 50,000+ = 2 (the two all-staff groups).
- Tier tables confirmed firing as designed across all 8 categories.

**Activities still pending:**

- Compare against the same Scenario 8 run at `Scale100k50Groups` to characterise where the cost differs.

**Validation criterion (still applies):** Run completes within the existing Scale100K time budget (< 2 hours). All Scenario 8 assertions pass.

## Success Criteria

1. All five renamed templates pass their existing scenario tests with no behavioural change
2. `Scale100k5kGroups` populates an OpenLDAP directory with 5,027 groups and approximately 750,000 memberships in under 15 minutes (initial estimate was ~900k; tier tables in `Get-LongTailGroupSize` lean toward smaller groups, so the measured total comes in lower while preserving the intended long-tail shape)
3. Full Scenario 8 run against `Scale100k5kGroups` completes within the existing Scale100K time budget (< 2 hours)
4. `Scale100k5kGroups` selection against Samba AD or non-Scenario-8 scenarios fails immediately with a clear error message
5. Documentation is consistent: no active docs reference the old template names
6. Group shape statistics for the new template show a long-tail distribution (at least 60% of groups under 100 members, no more than 1% over 10,000 members)

## Benefits

- **Self-describing naming:** Contributors can tell what a template provisions from its name alone
- **Realistic Scenario 8 test coverage:** Entitlement sync logic is now exercised against a group topology representative of real 100k-person organisations
- **Backward-compatible Samba coverage:** All existing scale tests continue to run on Samba AD; nothing is removed
- **Path to broader coverage:** Once validated on Scenario 8, the long-tail model can be extended to other scenarios in a follow-up

## Dependencies

- No new third-party packages
- No JIM-side code changes anticipated (Phase 5 will confirm)
- No infrastructure changes (OpenLDAP accesslog MDB is already sized to 8 GB, sufficient for the new template's write volume)

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| OpenLDAP accesslog MDB exhaustion at higher membership counts | Low | High (silent sync misses) | Current 8 GB allocation should hold; Phase 5 validates. If it doesn't, increase to 16 GB before merging |
| JIM-side performance regression with many small groups | Medium | Medium | Phase 5 measures explicitly; if regression appears, file as separate issue and decide whether to gate the new template behind the fix |
| Population time exceeds budget despite OpenLDAP optimisation | Medium | Medium | Phase 3 batching is the primary lever; if insufficient, fall back to fewer groups (~3,000) and re-tune size distribution |
| Snapshot rebuild cost for renamed templates | Low | Low | Snapshots are rebuilt on demand; orphaned snapshots are not breaking, just storage-wasteful. Document a cleanup command in the PR |
| Contributors invoke renamed templates by old name during transition | High | Low | `ValidateSet` errors are immediate and clear; no silent fallback. Changelog and release notes call out the rename |
| The five existing Scale tiers (200k-1M) have group counts in their names that imply intent the 200K+ tiers may want to revise later | Medium | Low | Out of scope for this change. The renames make those tiers' group counts explicit, which makes the future review easier, not harder |

## Decisions Recorded

- **Naming case:** lowercase `k` and `m` after the numeral (`Scale100k5kGroups`, `Scale1m80Groups`).
- **Snapshot/CSV cache cleanup:** no one-time script; orphaned images and cache entries are harmless and self-evict next time the user runs `docker image prune` or empties the CSV cache. Documented in the PR description rather than codified.
- **Layered naming convention** (for future scale tier reshapes):
  - Tier 1: base names from a curated pool
  - Tier 2: `{base}-{qualifier}` from a 5-entry qualifier pool
  - Tier 3: `{base}-{qualifier}-{N}` with a hyphen-separated serial
  Implemented for Department (`Get-DepartmentNames`) and for combinatorial categories via `Get-CombinatorialNames`.
